# Date Planner

Date Planner — веб-сервис для создания персональных приглашений на свидание и последующего совместного планирования. Сейчас проект поддерживает быстрый опубликованный сценарий, защищённые черновики расширенного режима, персональную интерактивную страницу для получателя, секретную ссылку автора и согласование одного из двух–пяти вариантов даты, времени и места.

## Стек

- **Frontend:** Vue 3, Nuxt, TypeScript, Composition API, ESLint и Vitest.
- **Backend:** Python, Django, Django REST Framework, PostgreSQL, django-cors-headers, drf-spectacular, pytest и Ruff.
- **Инфраструктура:** Docker, Docker Compose, GitHub Actions и конфигурация через переменные окружения.

## Требования

Для рекомендуемого локального запуска нужны только:

- Docker Engine с плагином Compose v2;
- Git;
- свободные порты `3000` и `8000`.

При работе через Docker Compose устанавливать Python, Node.js и PostgreSQL на хост не требуется.

## Первый запуск

Создайте локальный файл окружения из безопасного шаблона:

```bash
cp .env.example .env
```

Значения из `.env.example` предназначены только для локальной разработки. Для любого другого окружения обязательно замените секретный ключ и пароль базы данных.

Соберите и запустите все сервисы:

```bash
docker compose up --build
```

Backend дождётся готовности PostgreSQL и автоматически применит миграции. После добавления новой миграции её также можно применить вручную:

```bash
make migrate
```

Остановить стек можно командой `make down`. Данные PostgreSQL сохраняются в именованном томе `postgres_data`.

## API приглашений

Создать приглашение можно публичным запросом:

```bash
curl -X POST http://localhost:8000/api/v1/invitations/ \
  -H 'Content-Type: application/json' \
  -d '{
    "author_name": "Алиса",
    "recipient_name": "Борис",
    "message": "Давай сходим на свидание?",
    "creation_mode": "quick"
  }'
```

Поле `message` необязательно: его можно не передавать или отправить пустой строкой. Имена автора и получателя остаются обязательными. Поле `creation_mode` принимает `quick` или `extended`; если его не передать, backend сохранит `quick`.

Быстрый режим создаёт приглашение сразу со статусом `published` и заполняет `published_at`. Расширенный режим создаёт защищённый `draft`: management-ссылка работает сразу, а публичное чтение, ответ получателя и выбор варианта возвращают `404` до явной публикации.

Ответ содержит непредсказуемый UUID и `management_token`. Открытый токен возвращается только при создании: сервер сохраняет только его SHA-256-хеш. Сохраните сформированную frontend секретную ссылку автора — восстановить её позднее без аккаунта нельзя.

Получить публичные данные опубликованного приглашения можно по адресу:

```text
GET /api/v1/invitations/<uuid>/
```

Черновик остаётся доступен только автору через management endpoint. Опубликовать его можно идемпотентным запросом:

```http
PUT /api/v1/invitations/<uuid>/publish/
Authorization: Bearer <management-token>
```

Первый успешный запрос переводит приглашение в `published` и устанавливает `published_at`. Повторный `PUT` возвращает то же состояние и не меняет время публикации.

Получатель сохраняет свой ответ через публичную ссылку-возможность:

```http
PUT /api/v1/invitations/<uuid>/response/
Content-Type: application/json

{"response_status": "accepted"}
```

Допустимые ответы — `accepted` и `declined`. Повторный `PUT` позволяет изменить решение; `responded_at` хранит время последнего изменения ответа. Начальное состояние нового приглашения — `pending`.

Данные для страницы автора доступны по отдельному endpoint только с выданным токеном:

```text
GET /api/v1/invitations/<uuid>/manage/
Authorization: Bearer <management-token>
```

Тот же endpoint принимает частичное редактирование безопасных полей:

```http
PATCH /api/v1/invitations/<uuid>/manage/
Authorization: Bearer <management-token>
Content-Type: application/json

{
  "recipient_name": "Галина",
  "message": "Пойдём в театр?",
  "creation_mode": "extended"
}
```

Редактировать можно только `author_name`, `recipient_name`, `message` и `creation_mode`. Lifecycle-поля, management-токен, ответ получателя и данные планирования отклоняются с `400 Bad Request`, а не игнорируются. Пустой `PATCH` и повтор запроса с теми же нормализованными значениями идемпотентны и не меняют `updated_at`. Изменение черновика не публикует его; изменения опубликованного приглашения сразу видны по публичной ссылке.

После принятия приглашения автор сохраняет от двух до пяти вариантов свидания:

```http
PUT /api/v1/invitations/<uuid>/plan-options/
Authorization: Bearer <management-token>
Content-Type: application/json

{
  "options": [
    {
      "starts_at": "2026-08-01T18:30:00+07:00",
      "place": "Кофейня у парка",
      "comment": "Столик у окна"
    },
    {
      "starts_at": "2026-08-02T16:00:00+07:00",
      "place": "Набережная",
      "comment": ""
    }
  ]
}
```

Дата должна быть в будущем и содержать часовой пояс, место обязательно, а комментарий варианта можно оставить пустым. Пока получатель ничего не выбрал, повторный `PUT` атомарно заменяет набор вариантов. Будущий выбор блокирует изменение набора; исключение — неподтверждённый вариант, время которого уже прошло.

Получатель выбирает один из предложенных вариантов через публичную ссылку:

```http
PUT /api/v1/invitations/<uuid>/selection/
Content-Type: application/json

{"option_id": "<option-uuid>"}
```

Повтор того же выбора идемпотентен, а до итогового подтверждения получатель может выбрать другой вариант.

Автор фиксирует выбранный план отдельным необратимым запросом:

```http
PUT /api/v1/invitations/<uuid>/confirmation/
Authorization: Bearer <management-token>
Content-Type: application/json

{"confirmed": true, "option_id": "<selected-option-uuid>"}
```

Подтвердить можно только принятый, выбранный и ещё не наступивший вариант. UUID в запросе должен совпадать с вариантом, который видел автор: если получатель успел изменить выбор, сервер ничего не фиксирует и предлагает обновить данные. Повтор подтверждения того же варианта безопасен и не меняет временные метки. После него нельзя отклонить приглашение, выбрать другой вариант или заменить предложения; обе страницы показывают одинаковую финальную карточку плана.

Если неподтверждённый выбранный вариант успел пройти, frontend помечает его устаревшим, а автор может заменить набор будущими вариантами. Прошедшие варианты нельзя выбрать повторно. API также возвращает текущее серверное время: интерфейс сверяется с ним, поэтому неверные часы на устройстве не блокируют выбор или подтверждение.

Frontend создаёт две capability-ссылки:

- публичную `/invite/<uuid>` — она доступна только после публикации и отправляется получателю;
- секретную `/manage/<uuid>#token=<management-token>` — она работает и для черновика, и после публикации, её сохраняет автор.

Токен находится во fragment ссылки, поэтому браузер не отправляет его серверу вместе с первым HTTP-запросом. Страница считывает токен только в браузере, сразу очищает адресную строку и на время вкладки хранит его в `sessionStorage`; к API он передаётся в заголовке `Authorization`. В production обязателен HTTPS.

Каждое приглашение хранится отдельной записью, поэтому данные и ответы разных людей не перезаписывают друг друга. Публичного `GET /api/v1/invitations/` нет: API не раскрывает общий список приглашений. UUID публичной ссылки служит возможностью прочитать приглашение и отправить ответ, а отдельный management-токен — посмотреть данные и текущий статус на странице автора. Существующие до миграции management-токена записи остаются доступными только по публичной ссылке; при необходимости управления их следует создать заново.

Все API-запросы ограничены по частоте. По умолчанию анонимному клиенту разрешено `60` запросов в минуту, будущему авторизованному — `120`, создание приглашений дополнительно ограничено до `10` запросов в минуту с одного IP, а сохранение ответа и операции планирования — до `20` для каждого scope. При превышении сервер отвечает `429 Too Many Requests` и возвращает `Retry-After`.

В текущем Compose throttling использует общий файловый cache внутри одного backend-контейнера, поэтому лимит разделяется между Gunicorn workers. Это базовая защита от чрезмерного использования API, а не полноценная DDoS-защита. Перед горизонтальным production-развёртыванием понадобятся общий сетевой cache и rate limiting на уровне доверенного reverse proxy или API gateway.

## Тесты и проверки качества

Полный набор проверок запускается так:

```bash
make test
make lint
```

Отдельные проверки можно запустить явно:

```bash
docker compose run --rm backend pytest
docker compose run --rm backend ruff check .
docker compose run --rm frontend npm run test
docker compose run --rm frontend npm run lint
docker compose run --rm frontend npm run typecheck
docker compose run --rm frontend npm run build
```

GitHub Actions выполняет те же категории проверок backend и frontend при каждом push и pull request.

## Текущие возможности

- Адаптивная романтическая страница приглашения на локальных графических ресурсах.
- Управление ответами мышью, касанием и клавиатурой с учётом `prefers-reduced-motion`.
- Состояния принятого и отклонённого приглашения и кнопка сброса демонстрации.
- Выбор быстрого или расширенного режима создания с сохранением режима в PostgreSQL и API.
- Форма создания с клиентской и серверной валидацией, обработкой ошибок и копированием ссылок.
- Публичные персональные страницы приглашений по UUID через `/invite/<uuid>`.
- Секретные страницы автора по capability-ссылке без регистрации и пароля.
- Защищённое частичное редактирование имён, сообщения и режима с минимальным PATCH-payload, идемпотентными повторами и предупреждением о несохранённых изменениях.
- Четырёхшаговый каркас конструктора для расширенного черновика: приглашение, дата и время, активности и финал.
- Текущий шаг конструктора синхронизируется с URL и восстанавливается в пределах вкладки через `sessionStorage`.
- Первый шаг конструктора редактирует основные данные и автоматически сохраняет минимальный PATCH после паузы во вводе.
- Автосохранение выполняет только один запрос одновременно, ставит новые изменения в очередь, поддерживает ручной повтор и принудительное сохранение перед навигацией.
- Конструктор проверяет management-токен и не открывается для быстрого режима или уже опубликованного приглашения.
- Сохранение согласия или отказа получателя с состояниями загрузки, успеха и повтора после ошибки.
- Отображение актуального ответа и времени его получения на секретной странице автора.
- Редактор двух–пяти вариантов даты, времени и места на секретной странице автора.
- Выбор и изменение одного варианта получателем на публичной странице.
- Необратимое итоговое подтверждение автором и общая финальная карточка плана.
- Восстановление из состояния с истёкшим неподтверждённым вариантом.
- Жизненный цикл `draft` → `published` с серверным временем публикации и идемпотентным endpoint публикации.
- Расширенный режим создаёт приватный черновик; публичные recipient-endpoints скрывают его как отсутствующий.
- API создания, публикации, публичного чтения, сохранения ответа, планирования, подтверждения и защищённого чтения/редактирования автора через `/api/v1/`.
- Одноразовая выдача management-токена и хранение в базе только его хеша.
- Серверная валидация имён и текста приглашения без публичного списка записей.
- Общий и отдельные для создания, ответа и планирования rate limits с ответом `429 Too Many Requests`.
- Индикатор доступности backend, видимый только в режиме разработки.
- Health endpoint Django, OpenAPI-схема и Swagger UI.
- Локальная разработка с PostgreSQL и healthcheck сервисов.
- Автоматические линтеры, тесты, проверка типов и production-сборка frontend в CI.

## Ближайшие этапы

Следующий этап модуля конструктора — модель конфигурации отдельных экранов. Затем будут добавлены встроенная библиотека изображений, редактор первого экрана и интерактивное мобильное превью.

Регистрация на текущем этапе не нужна: секретная management-ссылка подтверждает право просмотра страницы автора. Аккаунты стоит добавлять только тогда, когда появятся личный кабинет, восстановление доступа, список нескольких приглашений одного автора или управление ими с разных устройств. Фоновые очереди и deployment-инфраструктуру также следует добавлять по мере конкретной необходимости.
