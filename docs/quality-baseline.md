# DPL-100: базовая линия качества

Эта контрольная точка фиксирует обязательные проверки перед развитием конструктора приглашений. Она не меняет пользовательский сценарий и схему базы данных.

## Что проверяется

### Backend

- Ruff lint и проверка форматирования;
- Django system checks;
- отсутствие изменений моделей без миграций;
- применение существующих миграций;
- генерация и валидация OpenAPI-схемы;
- pytest с branch coverage для пакета `apps.common`.

Из backend coverage исключены декларативные файлы `apps.py`, `urls.py` и миграции. Минимальное общее покрытие — `85%`.

### Frontend

- ESLint;
- Vitest;
- TypeScript-проверка Nuxt;
- production-сборка Nuxt;
- V8 coverage для `frontend/utils/**/*.ts`.

Текущий frontend-порог относится к типизированному слою бизнес-преобразований, который уже тестируется независимо от браузера:

| Метрика | Минимум |
| --- | ---: |
| Statements | 85% |
| Lines | 85% |
| Functions | 80% |
| Branches | 75% |

Vue-компоненты и browser-dependent composable-функции пока не входят в процентный порог. Их нельзя считать покрытыми только потому, что они импортируются. По мере появления component/E2E harness область coverage должна расширяться отдельными задачами, а существующие пороги нельзя снижать без обоснования.

## Локальный запуск

Сначала пересоберите образы после изменения зависимостей:

```bash
make build
```

Быстрые тесты без отчёта покрытия:

```bash
make test
```

Отчёты покрытия и пороговые проверки:

```bash
make coverage
```

Полная локальная проверка, эквивалентная категориям CI:

```bash
make quality
```

Отдельные команды:

```bash
make coverage-backend
make coverage-frontend
make lint
make check
```

## Артефакты

После локального запуска создаются игнорируемые Git файлы:

- `backend/coverage.xml`;
- `backend/htmlcov/index.html`;
- `frontend/coverage/index.html`;
- `frontend/coverage/coverage-summary.json`;
- `frontend/coverage/lcov.info`.

GitHub Actions сохраняет HTML/XML/LCOV отчёты как отдельные артефакты `backend-coverage` и `frontend-coverage`, даже если пороговая проверка завершилась ошибкой.

## Правило дальнейшей разработки

- новая бизнес-логика получает тесты в той же задаче;
- исправление дефекта получает regression-тест;
- область coverage постепенно расширяется, а не сужается ради прохождения CI;
- снижение порога допускается только после отдельного анализа и зафиксированного объяснения;
- процент покрытия не заменяет проверки переходов состояний, прав доступа, граничных значений и идемпотентности.
